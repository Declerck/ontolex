This document describes the specification of the lexicon model for ontologies as resulting from the work of the W3C ontology-lexicon Community Group.

The aim of the ontology-lexicon model is to provide rich linguistic grounding for ontologies. Rich linguistic grounding includes the representation of morphological and syntactic properties of lexical entries as well as the syntax-semantics interface, i.e. the meaning of these lexical entries with respect to an ontology.

This document is structured into eight sections, where the first five correspond to the main modules of the lexicon model for ontologies. Depending on their needs and requirements, applications will use one or more of the modules mentioned below, with the use of the ontolex module being the minimal choice.

  • Ontology-lexicon interface (ontolex)
  • Syntax and semantics (synsem)
  • Decomposition (decomp)
  • Variation and translation (vartrans)
  • Linguistic Metadata (lime)

The last three sections are not formally part of the actual specification of the model, but clarify :

  • how one can add linguistic levels of description by means of external ontologies (section 7)
  • how one can use the lexicon model for ontologies to describe lexical nets and other linguistic resources (section 8)
  • relation between the lexicon ontology model and SKOS (section 9)

Introduction

Ontologies are an important component of the Semantic Web. While ontology languages such as OWL allow us to define logical theories consisting of linguistically ungrounded symbols and corresponding axioms, a grounding in natural language is crucial in order to render such ontologies for human consumption and support meaningful interaction with them by human users. It seems reasonable to assume that access to the Semantic Web will be to a large extent mediated by natural language as this is the most natural means of human expression and communication.

Current web-based knowledge representations languages such as OWL and RDF(S) lack the rich linguistic grounding that is required for language-mediated access to ontologies. OWL and RDF(S) rely on a property rdfs:label to capture the relation between a vocabulary element and its (preferred) lexicalization in a given language. This lexicalization in some sense provides a lexical anchor that makes the concept, property, individual etc. understandable to a human user. The mechanisms for linguistic grounding available in OWL and RDF(S) can be seen at best as rudimentary. They are far from being able to capture the necessary linguistic and lexical information that Natural Language Processing (NLP) applications working with a particular ontology need. Such NLP applications are for example:

  • Natural language generation systems that produce coherent discourses by verbalizing a set of triples
  • Question Answering systems that interpret user questions with respect to one or more ontologies
  • Text interpretation systems that extract triples with respect to one or more ontologies
  • Query interpretation and semantic search in information retrieval systems
  • Natural language based interfaces to ontologies, Semantic Web and Linked Data.

Purpose of the model

The purpose of the model is to allow for linguistic grounding of a given ontology by adding information to the ontology about how the elements in the vocabulary of the ontology (individuals, classes, properties) are lexicalized in a given language.

The model follows the principle of semantics by reference1 in the sense that the semantics of a lexical entry is expressed by reference to an individual, class or property defined in the ontology. In some cases, the lexicon itself can add named concepts which are not made explicit in the ontology.

The model described here is supposed to be open in the sense that it provides a core vocabulary to add information about linguistic realisation of ontology and vocabulary elements. This vocabulary can and should be extended as required by a particular application. In particular, the model abstracts from the specific linguistic theory and category systems used to describe the linguistic properties of lexical entries and their syntactic behaviour, encouraging everyone to reuse existing data category systems or linguistic ontologies. The model is thus agnostic with respect to the linguistic theory and category systems adopted for the description of the linguistic properties. We make explicit in this document at which points to connect with an external repository of data categories or introduce own sub-properties of properties defined in the lexicon model for ontologies.

The model as presented here is inspired in many other models, in particular the Lexical Markup Framework (LMF), the LexInfo model, the LIR model, the Linguistic Meta Model (LMM), and the Senso Comune core model.

It is important to also mention what is not the purpose of the model:

  • It is not the goal of the model to replace any existing W3C Standard or Recommendation. In particular, this model is not intended to represent informal schemas such as taxonomies, thesauri and other classification schemes, such as the goal of the SKOS model.
  • It is not a vocabulary for annotation of texts. If you need to add annotations to textual data, then please consider using the Open Annotation, NLP Interchange Format (NIF) or the Extremely Annotational RDF Markup (Earmark).
  • It is not a formal, semantic model. The model is not supposed to be used to define an ontology and instead assumes that there is a given ontology in some ontology language that is to be linked to a lexicon that expresses how the classes, properties and individuals defined in the ontology are lexicalized.
  • It does not contain a complete collection of linguistic categories. There are many existing efforts to provide a vocabulary of description of properties of linguistic objects, such as ISOcat, OLiA, GOLD, LexInfo. The model recommends to build on those and does not introduce any vocabulary of linguistic description.
  • It is not a generic vocabulary for the publication of linguistic linked data. See the activities of the Open Linguistics Working Group for more information here.

Lexical Entries

The main class of the core of the lexicon ontology model is the class lexical entry. A lexical entry is defined as follows:

LexicalEntry (Class)

URI: http://www.w3.org/nl/lemon/ontolex/LexicalEntry

A single unit of analysis in the lexicon comprising a collection of forms that are morphologically related or have a single pronunciation and have the same set of meanings

SubClassOf: lexicalForm min 1 ontolex:Form, canonicalForm max 1 ontolex:Form, semiotics:Expression

A Lexical Entry thus needs to be associated with at least one form, which is defined below. Lexical entries are further specialized into Word, Affix (e.g., suffix, prefix or circumfix) and MultiwordExpression for example

:cat a ontolex:Word

:minimum_finance_lease_payments a ontolex:MultiwordExpression

:anti- a ontolex:Affix

Word (Class)

URI: http://www.w3.org/nl/lemon/ontolex/Word

A lexical entry consisting of a single word

SubClassOf: LexicalEntry

MultiwordExpression (Class)

URI: http://www.w3.org/nl/lemon/ontolex/MultiwordExpression

A lexical entry consisting of two or more words

SubClassOf: LexicalEntry

Affix (Class)

URI: http://www.w3.org/nl/lemon/ontolex/Affix

A lexical entry which is realized by joining as a prefix or similar to another word

SubClassOf: LexicalEntry

Forms

A lexical entry can be realized in different grammatical realizations. These different grammatical realizations are represented as different forms of the lexical entry in the core. A form is defined as follows:

Form (Class)

URI: http://www.w3.org/nl/lemon/ontolex/Form

The Form class represents one lexical variant of the written representation of a lexical entry.

SubclassOf: ontolex:writtenRep min 1 rdfs:Literal

A form can be connected to a lexical entry by means of the lexical form property, although it is preferred to use one of the two subproperties (canonical form, other form) defined below.

lexicalForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/lexicalForm

Indicates a single morphological form of an entry

Domain: LexicalEntry

Range: Form

Each form can thus have one or more written representations, defined as follows:

writtenRep (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/writtenRep

Indicates the written representation of a form

Domain: Form

Range: rdf:langString

SubPropertyOf: representation

As such a simple example of an entry may be as follows

<child> a ontolex:LexicalEntry ;                                                
  ontolex:lexicalForm <child#form_singular>, <child#form_plural> .              
                                                                                
<child#form_singular> a ontolex:Form ;                                          
  ontolex:writtenRep "child"@en .                                               
                                                                                
<child#form_plural> a ontolex:Form ;                                            
  ontolex:writtenRep "children"@en .

Different forms are used to express different morphological forms of the entry, they are no used to represent orthography, which are treated as different representations of the same form. A form should have at most one written representation per language tag, that is including region, script and other information, but all representations (and all forms) should be in the same language. For eaxample, for the lexical entry "colour", we would have two different representations of the same form, one for the British English written representation "colour" and one for the American English written representation. Both representations have the same pronunciation and the same meaning, so they are two different lexicographic variants of the same lexical entry:

:lex_color a ontolex:LexicalEntry;
     ontolex:form :form_color.

:form_color ontolex:writtenRep "colour"@en-GB, "color"@en-US.

A form may also have a phonetic representation, giving the phonetic pronunciation of the word.

phoneticRep (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/phoneticRep

Gives a phonetic representation of a form using a scheme such as the International Phonetic Alphabet

Domain: Form

Range: rdf:langString

SubPropertyOf: representation

The following example shows how we can represent two different pronunciations for one lexical entry using the example of "privacy" (the phonetic code is based on IPA):

:lex_privacy a ontolex:LexicalEntry;
     ontolex:form :form_privacy.

:form_privacy ontolex:writtenRep "privacy"@en;
     ontolex:phoneticRep "/ˈpɹɪv.ə.si/"@en-US-fonipa;
     ontolex:phoneticRep "/ˈpɹaɪ.və.si/"@en-GB-fonipa.

Each form should have exactly one written representation per language tag, but there are no restrictions on the number of associated phonetic representations. Phonetic representation and written representation are both considered to be subproperties of a more general property representation, for which users may define extra subproperties as required.

representation (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/representation

specifying a string by which the form is symbolized according to some orthographic scheme

Domain: Form

Range: rdf:langString

A lexical entry has a canonical form, which is the form that primarily identifies this entry and may be used as an index term in the lexicon. The canonical form for single words is typically the lemma of that word and is determined by lexicographic conventions for that language. In the case of verbs, the lemma is typically the infinitive form or, alternatively, the present tense of the verb (note that if an external particle is used to indicate the infinitive as in English "to play", this particle should be omitted). For nouns it is the noun singular form, while for adjectives it is the positive (i.e., non-negative, non-graded) form. For multi-word entries it is assumed that the same principles of lemmatization are applied to the head word.

The property canonical form has a Lexical Entry as domain and a form as range. It is a subproperty of the lexical form property. The canonical form has to be unique, thus the property canonical form is functional:

canonicalForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/canonicalForm

The canonical ("dictionary") form of the lexical entry. This usually indicates the "lemma" form of a lexical entry

Domain: LexicalEntry

Range: Form

Characteristics: Functional

SubPropertyOf: lexicalForm

It is recommended that in addition the canonical form is also given as the RDFS label of the lexical entry to enable indexing and interaction with existing systems. The lexical entries with the canonical form the noun "cat", the verb "marry" and the adjective "high" would look as follows (in Turtle syntax):

:lex_cat a ontolex:LexicalEntry, ontolex:Word;
     ontolex:canonicalForm :form_cat;
     rdfs:label "cat"@en.

:form_cat ontolex:writtenRep "cat"@en .

:lex_marry a ontolex:LexicalEntry, ontolex:Word;
     ontolex:canonicalForm :form_marry;
     rdfs:label "marry"@en.

:form_marry ontolex:writtenRep "marry"@en .

:lex_high a ontolex:LexicalEntry, ontolex:Word;
     ontolex:canonicalForm :form_high;
     rdfs:label "high"@en.

:form_high ontolex:writtenRep "high"@en .

Of course, lexical entries need not to correspond to one word only, they can correspond to a multi-word term, as the following example for the lexical entry "intangible assets" shows:

:lex_intangible_assets a ontolex:LexicalEntry, ontolex:MultiwordExpression ;
     ontolex:canonicalForm :form_intangible_assets.

:form_intangible_assets ontolex:writtenRep "intangible assets"@en .

We will see below how the lexical entry can be decomposed in its parts using the machinery introduced by the decomposition module.

It is also possible to indicate non-canonical forms of lexical entries which we call "other forms":

otherForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/otherForm

A non-preferred ("non-dictionary") representation of a lexical entry. This should be a variant that is either a morphological variant, an abbreviation, short form or acronym

Domain: LexicalEntry

Range: Form

SubPropertyOf: lexicalForm

For example we may specify non-canonical forms of the verb ""marry"" as follows:

:lex_marry a ontolex:LexicalEntry ;
  ontolex:canonicalForm ex:form_marry ;
  ontolex:otherForm ex:form_marries .

:form_marry ontolex:writtenRep "marry"@en .
:form_marries ontolex:writtenRep "marries"@en .

The morphological class (i.e., declension, conjugation or similar) may be specified with the morphological pattern property to avoid having to list all regular forms of a word. The implementation of these patterns is not specified by this document (but should be provided by some suitable vocabulary such as LIAM). For example:

:amare ontolex:morphologicalPattern :latin_first_conjugation ;
  ontolex:canonicalForm :amare_form .

:amare_form ontolex:writtenRep "amare"@la .

:videre ontolex:morphologicalPattern :latin_second_conjugation ;
  ontolex:canonicalForm :videre_form .

:videre_form ontolex:writtenRep "videre"@la

morphologicalPattern (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/morphologicalPattern

Indicates the morphological class of a word

Domain: LexicalEntry

Semantics

The model supports the specification of the meaning of lexical entries with respect to a given ontology. The lexicon model for ontologies follows the paradigm of semantics by reference2 in the sense that the meaning of a lexical entry is specified by pointing to the ontological concept that captures its meaning.

The property used for this is the property denotes, which is defined as the relation between a lexical entry and the logical predicate in an ontology that represents its meaning and has some denotational or model-theoretic semantics.

For the above mentioned lexical entries, this would look as follows:

:lex_cat a ontolex:LexicalEntry;
   ontolex:canonicalForm :form_cat;
   ontolex:denotes <http://dbpedia.org/resource/Cat>.
:form_cat ontolex:writtenRep "cat"@en.

:lex_marry a ontolex:LexicalEntry;
   ontolex:canonicalForm :form_marry;
   ontolex:denotes <http://dbpedia.org/ontology/spouse>.
:form_marry ontolex:writtenRep "marry"@en .

The property denotes is defined as follows:

denotes (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/denotes

The relation between a lexical entry and the logical predicate in an ontology that represents its meaning and has some denotational or model-theoretic semantics

Domain: ontolex:LexicalEntry

SubPropertyOf: semiotics:denotes

PropertyChain: sense o reference

The following example shows how we can model the fact that a certain lexical entry is ambiguous and assign it two (disjoint) meanings it can denote.

<bank> a ontolex:LexicalEntry ;
  ontolex:denotes <http://dbpedia.org/resource/Bank> ,
                  <http://dbpedia.org/resource/Bank_(geographic)> ;
  ontolex:canonicalForm <bank#lemma> .

<bank#lemma> ontolex:writtenRep "bank"@en .

No range is given for the denotes property, for two reasons. Firstly, the target of a denotation does not need to be an individual in the ontology but may also refer to a class, property or datatype defined by the ontology, as such it is not possible to specify the range of denotes in OWL. Secondly, the model is agnostic with respect to the ontology language used to express the ontology referred to, the assumption is is that the entity in the range represents some predicate that has a denotational semantics in some formal logical system.

Properties in the model for linking to ontologies have an inverse property named as "is x-ed by", where x is the original property name to enable the lexicon to be defined in an ontology focussed manner. In the case of denotes this property is isDenotedBy.

isDenotedBy (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/isDenotedBy

The inverse relation of #denotes</div><div class='description'>denotes

Lexical Sense & Reference

For many practical modelling situations, the denotes property is not sufficient to capture the precise linking between a lexical entry and its reference and as such we introduce an intermediate element called lexical sense to capture the sense(s) of a word which refers to the ontology entity. The lexical entry is linked to a lexical sense by means of the sense property and the lexical sense is linked to the ontology by means of the reference property. The chain sense ∘ reference is equivalent to the property denotes

LexicalSense (Class)

URI: http://www.w3.org/nl/lemon/ontolex/LexicalSense

Represents the lexical meaning of a lexical entry when interpreted as referring to the corresponding ontology element. A link between a lexical entry and an ontology entity via a LexicalSense object implies that the lexical entry can be used to refer to the ontology entity in question under corresponding connotations, pragmatic implications and meaning nuances that might no be directly modeled in the ontology. These pragmatic conditions and meaning nuances can be attached to the LexicalSense object, which represents a reification of the pair of LexicalEntry and ontology element. The Lexical Sense thus uniquely determines a lexical entry and an ontological entity it is associated with.

SubClassOf: ontolex:reference exactly 1 owl:Thing; ontolex:isSenseOf exactly 1 ontolex:LexicalEntry, semiotics:Meaning

Via the lexical sense object we can attach additional properties to a pair of lexical entry and ontological predicate that it denotes that describe under which conditions (context, register, domain, etc.) it is valid to regard the lexical entry as having the ontological entity as meaning. For example, we may wish to express the usages of the word "consumption" in terms of the topic and diachronic usage of the word. For this we will use the Dublin Core property subject and the LexInfo property dating as follows:

:lex_consumption a ontolex:LexicalEntry;
       ontolex:canonicalForm :form_consumption;
       ontolex:sense :consumption_sense1;
       ontolex:sense :consumption_sense2;
       ontolex:sense :consumption_sense3;
       ontolex:sense :consumption_sense4 .

:form_consumption ontolex:writtenRep "consumption"@en.

:consumption_sense1 dct:subject <http://dbpedia.org/resource/Ecology> ;
  ontolex:reference <http://dbpedia.org/resource/Consumption_(ecology)> .

:consumption_sense2 dct:subject <http://dbpedia.org/resource/Anatomy> ;
  ontolex:reference <http://dbpedia.org/Resource/Ingestion> .

:consumption_sense3 dct:subject <http://dbpedia.org/resource/Economics> ;
   ontolex:reference <http://dbpedia.org/resource/Consumption_(economics)> .

:consumption_sense4 dct:subject <http://dbpedia.org/resource/Medicine> ;
   lexinfo:dating lexinfo:old ;
   ontolex:reference <http://dbpedia.org/Resource/Tuberculosis> .

The lexical sense has a single lexical entry and a single reference in the ontology; thus properties "sense" and "reference" are defined as inverse functional and functional, respectively.

sense (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/sense

The relation between a lexical entry and one of its lexical senses

Domain: LexicalEntry

Range: LexicalSense

Characterstics: Inverse Functional

reference (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/reference

The relation between the lexical sense of a lexical entry and the ontological predicate that represents the denotation

Characteristics: Functional

Domain: LexicalSense

Their inverse relations are called isSenseOf and isReferenceOf:

isSenseOf (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/isSenseOf

Inverse property to #sense</div><div class='description'>sense

isReferenceOf (Object Property)

Inverse property to #reference</div><div class='description'>reference

Lexicons

Lexicon is an object that groups all lexical entries. A lexicon is specific for a particular language. The lexicon comprising the example entries we have created above would simply look as follows:

:lexicon a ontolex:Lexicon;
   ontolex:language "en";
   ontolex:entry :lex_high;
   ontolex:entry :lex_cat;
   ontolex:entry :lex_marry;
   ontolex:entry :lex_intangible_assets.

Lexicon (Class)

URI: http://www.w3.org/nl/lemon/ontolex/Lexicon

A collection of lexical entries specific to the given language and/or domain

SubClassOf: entry min 1, language exactly 1

The property linking a lexicon to a lexical entry is the property entry:

entry (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/entry

This property links a lexicon to one of the lexical entries contained in it.

Domain: Lexicon

Range: LexicalEntry

The language property can be stated on either a lexicon or a lexical entry and its value should be a String representing the language.

language (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/language

The language of a lexicon or an individual lexical entry.

Domain: Lexicon or LexicalEntry or void:Dataset

Range: rdf:langString

In addition, it is recommended that you use the Dublin Core language property with reference to either LexVo or [http://id.loc.gov/vocabulary/iso639-1.html The Library of Congress Vocabulary

  • LexVo codes should be of the form http://www.lexvo.org/page/iso639-3/xxx where xxx is the 3-Letter ISO 639-3 code
  • Library of Congress codes should be of the form http://id.loc.gov/vocabulary/iso639-1/xx where xx is the 2-Letter ISO 639-1 code

A full lexicon in Spanish for the above examples would be as follows:

:lexicon a ontolex:Lexicon;
   ontolex:language "es";
   ontolex:entry :lex_gato;
   ontolex:casar :lex_casar;
   ontolex:entry :lex_activo_intangible.

:lex_gato ontolex:canonicalForm :form_gato;
    ontolex:sense :gato_sense1;
    ontolex:sense :gato_sense2.

:gato_sense1 ontolex:reference <http://dbpedia.org/resource/Cat>.

:gato_sense2 ontolex:reference <http://dbpedia.org/resource/Jack_(device)>.

:form_alto ontolex:writtenRep "gato"@es.

:lex_casar ontolex:canonicalForm :form_casar;
    ontolex:sense :casar_sense1.

:casar_sense1 ontolex:reference <http://dbpedia.org/ontology/spouse>.

:form_casar ontolex:writtenRep "casar"@es.

:lex_activo_intangible ontolex:canonicalForm :form_activo_intangible;
    ontolex:sense :activo_intangible_sense1.

:activo_intangible_sense1 ontolex:reference <http://dbpedia.org/resource/Intangible_asset>.

:form_activo_intangible ontolex:writtenRep "activo intangible"@es .

This shows how the English lexicon and the Spanish lexicon are linked via the ontological references that the different words in the lexicon denote, thus integrating the lexica at the conceptual level.

Lexical Concept

Syntax and Semantics (synsem)

Lemon Syntax and Semantics.png

Syntactic Frames

Most words in a language do not stand by their own, but have a certain syntactic behaviour in the sense that they appear in certain structures and require a number of syntactic (and semantic arguments). Examples of this are transitive verbs (e.g. to own), which require a syntactic subject and a syntactic object as well as relational nouns (e.g. capital (of), mother (of), son (of), brother (of), etc.) which require a prepositional object or adjectives which require a noun to modify. Frames are defined as follows:

Frame (Class)

URI: http://www.w3.org/nl/lemon/synsem/Frame

A (syntactic) frame is a linguistic object that represents the syntactic behaviour of an open class word in terms of (syntactic) arguments it requires. It essentially describes the subcategorization structure of the open class word in question, in particular the (syntactic) arguments it requires.

In order to specify the syntactic behaviour of lexical entries in terms a number of possible (syntactic) frames, the synsem module defines the property syntactic behaviour.

synBehaviour (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/synBehavior

The property syntacticBehaviour relates a LexicalEntry to one of its possible syntactic behaviours as represented by a frame

Domain: LexicalEntry

Range: Frame

The following snippet shows how to indicate that the verb (to) own can be used as a transitive verb. This is accomplished by adding a frame own_fram that is declared as a transitive frame, using the class TransitiveFrame defined in the LexInfo Ontology.

:own_lex a ontolex:LexicalEntry ;
  ontolex:canonicalForm :own_form ;
  synsem:synBehavior :own_frame .

:own_frame a synsem:Frame, lexinfo:TransitiveFrame.

:own_form ontolex:writtenRep "own"@en .

A syntactic frame is defined by a number of (syntactic) arguments that need to be filled so that the frame is complete. A (syntactic) argument is defined as follows:

Argument (Class)

URI: http://www.w3.org/nl/lemon/synsem/Argument

An argument represents a slot that needs to be filled for a certain syntactic structure/projection of a lexical entry or a semantic frame to be complete. Syntactic arguments typically realise a certain grammatical function (e.g. subject, direct object, indirect object, prepositional object, etc.); semantic arguments represent an entity that plays some role in a semantic frame.

The object property synArg is used to relate a (syntactic) frame to one of its (syntactic) arguments.

synArg (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/synArg

The property synArg relates a Frame to one of its (syntactic) arguments.

Domain: Frame

Range: Argument

The following example shows how to extend the example for the verb (to) own by specifically indicating the arguments, in this case via two specific subproperties of synArg, i.e. lexinfo:subject or lexinfo:directObject defined in the external LexInfo ontology.

Note that if an external ontology is used to describe the type of arguments in more detail, e.g. indicating the grammatical function as in the example below, the external property used need to be made subproperties of synArg as in the example below.

:own_lex a ontolex:LexicalEntry ;
  synsem:canonicalForm :own_form ;
  synsem:synBehavior :own_synframe .

:own_form ontolex:writtenRep "own"@en. 

:own_synframe a lexinfo:TransitiveFrame;
       lexinfo:subject :own_frame_subj;
       lexinfo:directObject :own_frame_obj.

lexinfo:subject owl:subPropertyOf synsem:synArg.
lexinfo:directObject owl:subPropertyOf synsem:synArg.

Semantic Frames

From the semantic perspective, a (syntactic) frame corresponds to a semantic frame, defined as "as a coherent structure of related concepts that are related such that without knowledge of all of them, one does not have complete knowledge of any one; they are in that sense types of gestalt." Frames are represented by a coherent structure consisting of one or more predicates from a given ontology.

A semantic frame is thus defined as follows:

SemanticFrame (Class)

URI: http://www.w3.org/nl/lemon/synsem/SemanticFrame

A semantic frame is a coherent structure of related concepts that are related such that without knowledge of all of them, one does not have complete knowledge of any one; they are in that sense types of gestalt. The coherent structure is represented by one or more predicates from a given ontology.

subClassOf: ontolex:LexicalSense

In practice, semantic frames are the same as any other lexical sense and there is no reason to state that a sense is also a frame.

A semantic frame has (semantic) arguments that need to be filled for the frame to be complete. The property semArg relates a semantic frame to one of its semantic arguments.

semArg (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/semArg

Denotes a semantic argument slot of a semantic unit

Domain: SemanticFrame

Range: Argument

Conditions

The mapping of a word (lexical entry) to a specific meaning in the ontology is often complicated by pragmatic constraints on the usage in particular due to register and connotation of a word. For example we, may take the French words 'rivière' and 'fleuve' which refer to rivers flowing into a sea and flowing into other rivers respectively. Ideally, we should introduce specific ontology subclasses and link the meaning to this, however this may not be practical as it requires the client to make inferences to understand the word, and often capturing theses subtle differences is not possible using the existing named predicates in the ontology. As such, we can introduce conditions, which allow us to define sense that give a more or less general reference. These senses are not introduced instead of a formally defined sense but are there to help in application of the ontology-lexicon.

The details of conditions are mostly left to application specific vocabularies, but they could be given as natural langauge descriptions, e.g.,

:riviere a ontolex:LexicalEntry ;
  ontolex:sense :riviere_sense1 .

:fleuve a ontolex:LexicalEntry ;
  ontolex:sense : fleuve_sense1 .

:riviere_sense1 ontolex:reference :River ;
  synsem:condition [ 
    rdf:value "When flowing into the sea"@en
  ] .

:fleuve_sense2 ontolex:reference :River ;
  synsem:condition [
    rdf:value "When flowing into another river"@en
  ] .

condition (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/condition

Indicates a pragmatic restriction on the applicability of this sense

Domain: LexicalSense

Two special conditions are defined in this vocabulary, which consider pragmatic restrictions on the range or domain of a property. This is to capture the common case where the selection of verb depends on the subject or object of that verb. For example, in English a distinction is made between the verbs used to express operating a machine for transportation, that is in English one rides a bike but drives a car, a distinction that is not found in many other languages. We may not wish to introduce specific properties for these cases but can instead do it with a lexical selection constraint as follows:

:ride a ontolex:LexicalEntry ;
  ontolex:sense :ride_sense1 .

:ride_sense1 ontolex:reference :methodOfTransportation ;
  ontolex:propertyRange :Bicycle .

Decomposition (decomp)

Lemon Decomposition.png

Decomposition is the process of indicating which elements constitute a multi-word or compound lexical entry. The simplest way to do this is by means of subterm property, which indicates that a lexical entry is a part of another entry.

subterm (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/subterm

Indicates that one lexical entry is a constituent part of another lexical entry

Domain: LexicalEntry

Range: LexicalEntry

For example we may indicate that "landing" is a part of the lexical entry "amphibious landing" as follows:

:AmphibiousLanding a ontolex:LexicalEntry ;
  decomp:subterm :Landing.

The use of sub-terms has two principle limitations, firstly it is not possible to indicate inflection within a particular phrase of its individual elements, and secondly that it is not clear if the components represent the complete phrase or their relative order. To fix this we introduce an object called a component defined as follows:

Component (Class)

URI: http://www.w3.org/nl/lemon/decomp/Component

A component represents a part of a lexical entry, that is either a word, sequence of words, phrase, or affix

Each component is said to be a constituent of a lexical entry

Constituent (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/constituent

Indicates a component that is part of an entry

Domain: LexicalEntry or Component

Range: Component

Furthermore, each of these components is said to identify a lexical entry, that gives the link to the word itself

Identifies (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/identifies

Indicates that the component represents the usage of this particular lexical entry within another lexical entry

Domain: Component

Range: LexicalEntry or Argument

As such we may represent simple decompositions such as tokenizations as follows

:DesembarcoAnfibio a ontolex:LexicalEntry ;

:DesembarcoAnfibio_Anfibio_Component a decomp:Component ;
  lexinfo:gender lexinfo:masculine ;
  lexinfo:number lexinfo:plural ;
  decomp:identifies :Anfibio .

  decomp:constituent :AfricanSwineFever_African_Component :
  decomp:constituent :SwineFever_Component .

TerminologicalVariant (Class)

URI: http://www.w3.org/nl/lemon/vartrans/TerminologicalVariant

The variants vary along dimensions that are not captured by the given ontology and are intentionally (pragmatically) caused.

SubclassOf: vartrans:source exactly 1 ontolex:LexicalSense; vartrans:target exactly 1 ontolex:LexicalSense; vartrans:Variant

Examples of terminological variants are:

  • Diatopic (dialectal or geographical variants) (e.g., gasoline vs. petrol)
  • Diaphasic (register) (e.g., headache and cephalalgia; swine flu and pig flu and H1N1 and Mexican pandemic flu)
  • Diachronic (or chronological variants) (e.g., tuberculosis and phthisis)
  • Diastratic (discursive or stylistic vari‐ants) (e.g., man vs. bloke)
  • Dimensional variants: the terms point to the same concept but highlight a different property or dimension of the concept (e.g., biosanitary waste vs. hospital waste; Novel Coronavirus vs. Middle East Respiratory Syndrome Coronavirus; obsolete technology vs. dangerous technology; madre de alquiler (rental mother) vs. vientre de alquiler (womb mother), in Spanish).

Translation

A special type of variant is a variant that relates lexical senses across languages. We call such a variant a translation. Special types of such translations are "direct equivalents" (representing terms that describe (highly) semantically equivalent entities that refer to entities that exist in both cultures and languages. E.g. surrogate mother in English, madre de alquiler in Spanish and mère porteuse in French.) and "cultural equivalents" (representing the closest equivalent -or counterpart- of a sense in another language. Eg. Prime Minister, Bundeskanzler). Such translation categories are not included in the model and will be referred to external catalogs by means of the variation category. As such a Translation is defined as follows:

Translation (Class)

URI: http://www.w3.org/nl/lemon/vartrans/Translation

Two lexical senses in different languages stand in a translation relation to each other if they are equivalent in the sense that they can be exchanged for each other without any meaning implications.

SubclassOf: vartrans:TerminologicalVariant; ontolex:source exactly 1 ontolex:LexicalSense ontolex:target exactly 1 ontolex:LexicalSense;

As an example of a rather literal translation we give the following: "surrogate mother" in English, "madre de alquiler" in Spanish and "mère porteuse" in French; and cultural qquivalents: "prime minister" in English and "Bundeskanzler" in German. These are represented as follows:

:surrogate_mother a ontolex:LexicalEntry;
      ontolex:language "en";
      ontolex:sense :surrogate_mother_sense.

:surrogate_mother_sense ontolex:reference <http://dbpedia.org/resource/Surrogacy>.

:madre_de_alquiler a ontolex:LexicalEntry;
      ontolex:language "es";
      ontolex:sense :madre_de_alquiler_sense.

:madre_de_alquiler_sense ontolex:reference <http://es.dbpedia.org/resource/Madre_de_alquiler>.

:leihmutter a ontolex:LexicalEntry;
      ontolex:language "de";
      ontolex:sense :leihmutter_sense.

:surrogate_mother_sense ontolex:reference <http://de.dbpedia.org/resource/Leihmutter>.

:trans1: a vartrans:Translation;
vartrans:source :surrogate_mother_sense;
vartrans:target :madre_de_alquiler_sense;
vartrans:variationCategory :literalTranslation.

Lexicalization

One of the main elements that is of interest to record is how completely a lexicon covers the vocabulary of a given ontology for the language of the lexicon, and to this end a lexicon may be associated with a Lexicalization Set object which defines the lexicalization set of the ontology with this particular lexicon. The lexicalization set object should be unique for a given lexicon-ontology pair.

LexicalizationSet (Class)

URI: http://www.w3.org/nl/lemon/lime/LexicalizationSet

A lexicalization set provides the realization in a natural language of the resources found in an RDF dataset/ontology vocabulary. To that end, the lexicalizations present in the set might relate these resources to entries of a Lexicon (i.e., lexical entries or senses). Multiple linguistic models may coexist in principle, as long as the same information is expressed in each model, in the limit of its expressive power.

SubClassOf: lexicalizedDataset exactly 1 and linguisticModel exactly 1 and lexicon exactly 1 and language exactly 1

SubClassOf: void:Dataset

The lexicalization set is linked to the ontology and lexicon object by means of the properties reference dataset and lexicon dataset respectively.

referenceDataset (Object Property)

URI: http://www.w3.org/nl/lemon/lime/referenceDataset

The void:Dataset that stands for the data being linguistically grounded by a lime:Lexicalization.

Domain: LexicalizationSet

Range: void:Dataset

lexiconDataset (Object Property)

URI: http://www.w3.org/nl/lemon/lime/lexiconDataset

The lime:Lexicon a lime:Lexicalization refers to.

Domain: LexicalizationSet

Range: Lexicon

In the lexicalization we may give the number of references lexicalized in the ontology with the eponymous property as well as the number of entries used to lexicalize. These numbers may be lower than the number of entites in the ontology or entries in the lexicon respectively.

URI: http://www.w3.org/nl/lemon/lime/references

Linguistic Description

One of the most important aspects for a lexicon is recording the linguistic types of words in the lexicon. However, the types used in these lexicons are often very different and as such, it is not possible to provide in advance a complete categorization of all the possible categories. Instead we will describe the application of a specific data category, that is [http://lexinfo.net/ontology/2.0/lexinfo# LexInfo] as a data category system.

Morphosyntactic Description

Perhaps the most common description found in a lexicon is the part-of-speech, which may be carried shown using LexInfo as follows

<cat> a ontolex:Word ;
  lexinfo:partOfSpeech lexinfo:noun .

When defining a data category selection it is important to give good descriptions as well as links to other models if possible. For example, LexInfo defines partOfSpeech and noun as:

lexinfo:partOfSpeech 
  rdfs:label "part of speech"@en ;
  rdfs:comment "A category assigned to a word based on its grammatical and semantic properties."@en ;
  dcr:datcat <http://www.isocat.org/datcat/DC-1345> ,
             <http://www.isocat.org/datcat/DC-396> ;
  rdfs:range lexinfo:PartOfSpeech ;
  rdfs:subPropertyOf lexinfo:morphosyntacticProperty .

lexinfo:noun
  a lexinfo:PartOfSpeech, lexinfo:NounPOS ;
  rdfs:label "noun"@en ;
  rdfs:comment "Part of speech used to express the name of a person, place, action or thing."@en ;
  dcr:datcat <http://www.isocat.org/datcat/DC-1333> .

This way LexInfo provides a label, description, link to ISOcat as well as axioms limiting the range of values.

The following morphosyntactic properties are defined in LexInfo

  • Animacy: Whether a word is animate (human or animal), or agreeing form of a verb or adjective
  • Aspect: Grammatical aspect, especially for perfect and imperfect verbs
  • Case: Case of a noun, or case agreement of adjectives
  • Cliticness: Whether the word acts as a clitic
  • Definitiness: Generally of articles
  • Degree: Comparative or superlative forms of adjectives
  • Finiteness: Whether a form is finite
  • Gender: Grammatical gender of a noun, or agreeing form of a verb or adjective
  • Modification Type: Whether modification is before or after the modified word, e.g., a proclitic has cliticness and is a premodifier
  • Mood: The modality (e.g., imperative, conditional) of a verb
  • Negative: Negative forms of verbs, etc., e.g., in Japanese
  • Number: Marks the plural or dual of a noun, or agreeing form of a verb or adjective
  • Part of speech: The syntactic class of a word
  • Person: Whether a noun or pronoun refers to the speaker (first person) or listener (second person) or other entity (third person) and agreeing forms of verbs
  • Referent Type: Whether an article, pronoun or similar is a possessive form
  • Tense: The temporal usage of a verb or adjective
  • Verb Form Mood: --John McCrae (talk) 15:16, 23 January 2015 (UTC) Merge with mood?
  • Voice: Indicates a passive form of a verb

When using these properties care should be taken to distinguish between annotations on the form and annotations on the entry. It should be assumed that all annotations on a entry also apply to all forms. For example, in many languages gender is an entry property for nouns, but a form property for adjectives, for example:

<spiaggia> a ontolex:Word ;
  ontolex:canonicalForm <spiaggia#lemma> ;
  ontolex:otherForm <spiaggia#plural> ;
  lexinfo:partOfSpeech lexinfo:noun ;
  lexinfo:gender lexinfo:feminine .

<spiaggia#lemma> 
  ontolex:writtenRep "spiaggia"@it ;
  lexinfo:number lexinfo:singular .

<spiaggia#plural>
  ontolex:writtenRep "spiagge"@it ;
  lexinfo:number lexinfo:plural .

<famoso> a ontolex:Word ;
  ontolex:canonicalForm <famoso#lemma> ;
  ontolex:otherForm <famoso#famosa>, <famoso#famose>, <famoso#famosi> ;
  lexinfo:partOfSpeech lexinfo:adjective .

<famoso#lemma> 
  ontolex:writtenRep "famoso"@it ;
  lexinfo:number lexinfo:singular ;
  lexinfo:gender lexinfo:masculine .

<famoso#famosa> 
  ontolex:writtenRep "famosa"@it ;
  lexinfo:number lexinfo:singular ;
  lexinfo:gender lexinfo:feminine .

For convenience, LexInfo also provides classes for parts of speech to replace using Word, for example Noun is defined as:

Noun ≡ ∃ partOfSpeech.NounPOS

We recommend that you use this along side property based annotations to ensure maximum compatibility, e.g.,

<geneesmiddel> a lexinfo:Noun ;
  lexinfo:partOfSpeech lexinfo:noun .

Pragmatic Description

As with morphosyntactic description, there are a number of properties that can be used to describe the pragmatic usage of a term. These terms should either be attached to the lexical sense or the concept. The following sense properties are commonly used

  • Dating: The temporal usage of a term (e.g., if not modern)
  • Normative Authorization: Any control on the usage of the term by standards bodies
  • Register: The formality level or similar when a word should be used

In addition there are many relationships which are generally defined between concepts in the lexicon

  • Approximate(Synonym)
  • Antonym
  • Casually Related
  • Holonym
  • Hypernym
  • Hyponym
  • Meronym
  • Pertains to
  • Quasi-equivalent
  • Related
  • Synonym

Arguments

When describing syntactic frames it is necessary to state the roles played by different arguments. For example we may wish to indicate which of an argument is the subject and which is the direct object as has been described previously. LexInfo defines the following types of arguments

  • Subject
  • Object: Subclassed into direct, indirect, prepositional (a non-optional object marked with a preposition) and gentive
  • Adjunct: Optional arguments subclassed into prepositional, possessive, comparative and superlative
  • Copulative: The subject or object of the copula, copulative subject is used if the argument can only be the subject of the copula
  • Clausal: An argument that is realised as a clause may be declarative, gerundive, infinitive, interrogative (infinitive), possessive infinitive, prepositional gerund/interrogative, sentential or subjunctive
  • Attributive: The word modified by an an adjective in the attributive pseudo-frame

Each argument is associated with a property and a class with the appropriate name, for example:

<father> a lexinfo:Noun ;
  synsem:synBehavior <father#frame> .

<father#frame> a lexinfo:NounPredicateFrame ;
  rdfs:label "X is the father of Y" , "X is Y's father" ;
  lexinfo:copulativeArg <father#arg1> ;
  lexinfo:possessiveAdjunct <father#arg2> .

<father#arg1> a lexinfo:CopulativeArg .

<father#arg2> a lexinfo:PossessiveAdjunct .

Frames

Frames are defined by the arguments that they have, for example a frame with a subject and a direct object is called a transitive frame.

In LexInfo, such frames are axiomatized as follows:

   TransitiveFrame ≡ VerbFrame ⊓ (=1 subject ⊓ =1 directObject)

In some special cases it may be possible that the frame encodes information about the that is not simply the argument, for example LexInfo contains two frames DitransitiveFrame_For and DitransitiveFrame_To both of which encode verbs that take two objects in English, however also have a form where they have only a single direct object and a prepositional phrase object, e.g.,

X gives Y Z/X gives Z to Y

X buys Y Z/X buys Z for Y

Lexical Nets

Lexical nets, in particular wordnets, are one of the most important forms of lexical data, and we briefly describe how a wordnet may be represented on the web using the OntoLex/Lemon vocabulary.

WordNet in OntoLex/Lemon

In general, the following items as termed by WordNet are considered to be equivalent to the following classes in OntoLex/Lemon

  • Synset: Lexical Concept
  • Word: Lexical Entry
  • (Word) Sense: Lexical Sense
  • Lemma: Canonical Form

Other properties such as hypernym may be defined separately, such as is done by the WordNet-RDF ontology available at

   http://wordnet-rdf.princeton.edu/ontology

For example, here is a simple modelling example:

<> a ontolex:LexicalEntry ;
    wordnet-ontology:part_of_speech wordnet-ontology:noun ;
    ontolex:canonicalForm <#CanonicalForm> ;
    ontolex:sense <#1-n>,
        <#2-n>,
        <#3-n>,
        <#4-n>,
        <#5-n>,
        <#6-n> .

<#1-n> a ontolex:LexicalSense ;
    wordnet-ontology:gloss "feline mammal usually having thick soft fur and no ability to roar: domestic cats; wildcats"@eng ;
    wordnet-ontology:lex_id 0 ;
    wordnet-ontology:old_sense_key "cat%1:05:00::" ;
    wordnet-ontology:sense_number 1 ;
    wordnet-ontology:tag_count 18 ;
    ontolex:reference <http://wordnet-rdf.princeton.edu/wn31/102124584-n> ;
    owl:sameAs <http://lemon-model.net/lexica/uby/wn/WN_Sense_574>,
        <http://www.lexvo.org/page/wordnet/30/noun/cat_1_05_00> .

<http://wordnet-rdf.princeton.edu/wn31/102124584-n> a ontolex:LexicalConcept ;
    rdfs:label "cat"@eng,
        "true cat"@eng ;
    wordnet-ontology:gloss "feline mammal usually having thick soft fur and no ability to roar: domestic cats; wildcats"@eng ;
    wordnet-ontology:hypernym <http://wordnet-rdf.princeton.edu/wn31/102123961-n> ;
    wordnet-ontology:hyponym <http://wordnet-rdf.princeton.edu/wn31/102124772-n>,
        <http://wordnet-rdf.princeton.edu/wn31/102127587-n> ;
    wordnet-ontology:lexical_domain wordnet-ontology:noun.animal ;

Relation to Other Models

This part of the document is not part of the lexicon ontology model proper, but contains concrete recommendations and best practice patterns for using Ontolex/Lemon with other models.

SKOS(-XL)

The use case we adress is one where a thesaurus or other taxonomic resource or classification system in SKOS needs to be enriched with more detailed linguistic information.

Imagine we have two concepts "financial assets" and "tangible assets" modeled in a SKOS. The official way to link a lexical entry to a SKOS concept (and viceversa) is through the property evokes (and its inverse isEvokedBy). The following example shows how to do this:

:financial_assets a skos:Concept;
                ontolex:isLexicalizedBy :financial_assets_lex.

:financial_assets_lex a ontolex:LexicalEntry;
                 ontolex:evokes :financial_assets;
                 ontolex:form :financial_assets_form. 

:financial_assets_form ontolex:writtenRep "financial assets".

The above represents the standard way of linking a SKOS concept to a lexical entry in the lexicon ontology model.

To show how to make statements about preferred lexicalizations akin to the properties prefLabel, altLabel and hiddenLabel as used in SKOS, we give the following example:

:tuberculosis a skos:Concept;
     ontolex:isEvokedBy :tuberculosis_lex;
     ontolex:isEvokedBy :consumption_lex.

:tuberculosis_lex a ontolex:LexicalEntry;
      ontolex:sense :tuberculosis_sense;
      ontolex:evokes :tuberculosis.
   
:tuberculosis_sense a ontolex:LexicalSense;
      ontolex:isContainedIn :tuberculosis; 
      ontolex:context "preferred".

:consumption_lex a ontolex:LexicalEntry;
       ontolex:sense :consumption_sense;
       ontolex:evokes :tuberculosis.

:consumption_sense a ontolex:LexicalSense;
        ontolex:isContainedIn :tuberculosis;
        ontolex:context "outdated".

In case you are using reified labels as in SKOS-XL, it is possible to have Forms or LexicalEntries in the range of the skosxl:prefLabel, skosxl:altLabel and skosxl:hiddelLabel properties. However, we note that from this it would follows that LexicalEntries and Forms would be inferred to be skosxl:Labels, which does not correspond to the understanding of Forms and LexicalEntries of this community as linguistic objects rather than mere `labels'.

LMF

The Lexical Markup Framework (ISO-24613:2008) is a standard for representing lexicons, however is currently not an open standard and thus we do not recommend using it to publish data on the web. LMF has however provided a key inspiration to the creation of the OntoLex/LMF model. As such some classes correspond nearly exactly to their correspondant in OntoLex/Lemon, e.g., LexicalEntry. A fuller description of the relationship between LMF and the original lemon model is provided here, however the relationship can be briefly summarized as follows

  • OntoLex/Lemon defines the meaning of a term by reference to an ontology element defined by the OWL model.
  • OntoLex/Lemon has a much simpler usage, removing many elements that were in LMF
  • OntoLex/Lemon uses ontologies of linguistic description such as LexInfo to annotate its elements
  • OntoLex/Lemon does not have the following
    • Morphology: See the LIAM model for an alternative
    • Global Lexicon Constraints: These can be stated using the OWL vocabulary if necessary

OpenAnnotation and NIF

A common use case of a lexicon model is to annotate text as it occurs in a document, for example by indicating the sense, synset (concept) or entry (lemma) that is being referred to. In this section we will consider how these links can be made using two emerging standards Open Annotation and the NLP Interchange Format (NIF). In both models an element of lexicon may be the target of an annotation. This target may be a Form, Lexical Entry, Lexical Sense or LexicalConcept and it is important to give the class to make clear what the target of the annotation is.

For both models, we will now give an example of annotating a word "cat" occurring at character 7 in a file at the URL 3.

In NIF the connection is created by means of the nif:occurrence property, whose subject should be the OntoLex/Lemon element. An example is given below:

@prefix nif: <http://persistence.uni-leipzig.org/nlp2rdf/ontologies/nif-core#> .
@prefix ontolex: <http://www.w3.org/ns/lemon/ontolex#> .

<http://www.example.com/doc.txt#char=7,10> a nif:RFC5147String ;
  nif:beginIndex 7 ;
  nif:endIndex 10 ;
  nif:anchorOf "cat" .

<cat> a ontolex:LexicalEntry ;
  nif:occurrence <http://www.example.com/doc.txt#char=7,10> .

In OpenAnnotation, the OntoLex/Lemon element is given as the body of an annotation. For example

@prefix dctypes: <http://purl.org/dc/dcmitype/> .
@prefix oa: <http://www.w3.org/ns/oa#> .
@prefix ontolex: <http://www.w3.org/ns/lemon/ontolex#> .

<anno> a oa:Annotation ;
  oa:hasBody <cat> ;
  oa:hasTarget <anno#target> .

<anno#target> a dctypes:Text ;
  oa:hasSelector <http://www.example.com/doc.txt#char=7,10> .

<http://www.example.com/doc.txt#char=7,10> a oa:FragmentSelector .

<cat> a ontolex:LexicalEntry .
