This document describes the specification of the lexicon model for ontologies as resulting from the work of the W3C ontology-lexicon Community Group.

The aim of the ontology-lexicon model is to provide rich linguistic grounding for ontologies. Rich linguistic grounding includes the representation of morphological and syntactic properties of lexical entries as well as the syntax-semantics interface, i.e. the meaning of these lexical entries with respect to an ontology.

This document is structured into eight sections, where the first five correspond to the main modules of the lexicon model for ontologies. Depending on their needs and requirements, applications will use one or more of the modules mentioned below, with the use of the ontolex module being the minimal choice.

  • Ontology-lexicon interface (ontolex)
  • Syntax and semantics (synsem)
  • Decomposition (decomp)
  • Variation and translation (vartrans)
  • Linguistic Metadata (lime)

The last three sections are not formally part of the actual specification of the model, but clarify :

  • how one can add linguistic levels of description by means of external ontologies (section 6)
  • how one can use the lexicon model for ontologies to describe lexical nets and other linguistic resources (section 7)
  • relation between the lexicon ontology model and SKOS (section 8)

Introduction

Ontologies are an important component of the Semantic Web. While ontology languages such as OWL allow us to define logical theories consisting of linguistically ungrounded symbols and corresponding axioms, a grounding in natural language is crucial in order to render such ontologies for human consumption and support meaningful interaction with them by human users. It seems reasonable to assume that access to the Semantic Web will be to a large extent mediated by natural language as this is the most natural means of human expression and communication.

Current web-based knowledge representations languages such as OWL and RDF(S) lack the rich linguistic grounding that is required for language-mediated access to ontologies. OWL and RDF(S) rely on a property rdfs:label to capture the relation between a vocabulary element and its (preferred) lexicalization in a given language. This lexicalization in some sense provides a lexical anchor that makes the concept, property, individual etc. understandable to a human user. The mechanisms for linguistic grounding available in OWL and RDF(S) can be seen at best as rudimentary. They are far from being able to capture the necessary linguistic and lexical information that Natural Language Processing (NLP) applications working with a particular ontology need. Such NLP applications are for example:

  • Natural language generation systems that produce coherent discourses by verbalizing a set of triples
  • Question Answering systems that interpret user questions with respect to one or more ontologies
  • Text interpretation systems that extract triples with respect to one or more ontologies
  • Query interpretation and semantic search in information retrieval systems
  • Natural language based interfaces to ontologies, Semantic Web and Linked Data.

Purpose of the model

The purpose of the model is to allow for linguistic grounding of a given ontology by adding information to the ontology about how the elements in the vocabulary of the ontology (individuals, classes, properties) are lexicalized in a given language.

The model follows the principle of semantics by reference in the sense that the semantics of a lexical entry is expressed by reference to an individual, class or property defined in the ontology. In some cases, the lexicon itself can add named concepts which are not made explicit in the ontology.

The model described here is supposed to be open in the sense that it provides a core vocabulary to add information about linguistic realisation of ontology and vocabulary elements. This vocabulary can and should be extended as required by a particular application. In particular, the model abstracts from the specific linguistic theory and category systems used to describe the linguistic properties of lexical entries and their syntactic behaviour, encouraging everyone to reuse existing data category systems or linguistic ontologies. The model is thus agnostic with respect to the linguistic theory and category systems adopted for the description of the linguistic properties. We make explicit in this document at which points to connect with an external repository of data categories or introduce own sub-properties of properties defined in the lexicon model for ontologies.

The model as presented here is inspired in many other models, in particular the Lexical Markup Framework (LMF), the LexInfo model and the LIR model.

It is important to also mention what is not the purpose of the model:

  • It is not the goal of the model to replace any existing W3C Standard or Recommendation. In particular, this model is not intended to represent informal schemas such as taxonomies, thesauri and other classification schemes, such as the goal of the SKOS model.
  • It is not a vocabulary for annotation of texts. If you need to add annotations to textual data, then please consider using the NLP Interchange Format (NIF).
  • It is not a formal, semantic model. The model is not supposed to be used to define an ontology and instead assumes that there is a given ontology in some ontology language that is to be linked to a lexicon that expresses how the classes, properties and individuals defined in the ontology are lexicalized.
  • It does not contain a complete collection of linguistic categories. There are many existing efforts to provide a vocabulary of description of properties of linguistic objects, such as ISOcat, OLiA, GOLD, LexInfo. The model recommends to build on those and does not introduce any vocabulary of linguistic description.
  • It is not a generic vocabulary for the publication of linguistic linked data. See the activities of the Open Linguistics Working Group for more information here.

Core

Lexical Entries

The main class of the core of the lexicon ontology model is the class lexical entry. A lexical entry is defined as follows:

LexicalEntry (Class)

URI: http://www.w3.org/nl/lemon/ontolex/LexicalEntry

A single unit of analysis in the lexicon comprising a collection of forms that are morphologically related or have a single pronunciation and have the same meaning(s).

SubClassOf: lexicalForm min 1 ontolex:Form, canonicalForm max 1 ontolex:Form, semiotics:Expression

A Lexical Entry thus needs to be associated with at least one Form, which is defined in the following.

Forms

A lexical entry can be realized in different grammatical realizations. These different grammatical realizations are represented as different Forms of the lexical entry in the core. A Form is defined as follows:

Form (Class)

URI: http://www.w3.org/nl/lemon/ontolex/Form

The Form class represents one lexical variant of the written representation of a lexical entry.

SubclassOf: ontolex:writtenRep min 1 rdfs:Literal

Each form can thus have one or more written representations, defined as follows:

writtenRep (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/writtenRep

Indicates the written representation of a form

Domain: Form

Range: rdf:langString

SubPropertyOf: representation

For example, for the lexical entry "colour", we would have two different forms, one for the British English written representation "colour" and one for the American English written representation. Both representations have the same pronunciation and the same meaning, so they are two different lexicographic variants of the same lexical entry:

:lex_colour a ontolex:LexicalEntry;
         ontolex:form :form_colour;
         ontolex:form :form_color.
    
    :form_colour ontolex:writtenRep "colour"@en-GB.
    :form_color ontolex:writtenRep "color"@en-US.

It can also have different phonetic representations corresponding to different ways of pronouncing the form.

phoneticRep (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/phoneticRep

Gives a phonetic representation of a form using a scheme such as the International Phonetic Alphabet

Domain: Form

Range: rdf:langString

SubPropertyOf: representation

The following example shows how we can represent two different pronunciations for one lexical entry using the example of "privacy" (the phonetic code is based on IPA):

:lex_privacy a ontolex:LexicalEntry;
         ontolex:form :form_privacy.
    
    :form_privacy ontolex:writtenRep "privacy"@en;
         ontolex:phoneticRep "/ËpÉ¹Éªv.É.si/"@en-US-fonipa;
         ontolex:phoneticRep "/ËpÉ¹aÉª.vÉ.si/"@en-GB-fonipa.

Each form should have exactly one written representation per language tag, but there are no restrictions on the number of associated phonetic representations. Phonetic representation and written representation are both considered to be subproperties of a more general property representation, for which users may define extra subproperties as required.

representation (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/representation

specifying a string by which the form is symbolized according to some orthographic scheme

Domain: Form

Range: rdf:langString

A lexical entry can have a canonical form, which is the form that primarily identifies this entry and may be used as an index term in the lexicon. The canonical form for single words is typically the lemma of that word and is determined by lexicographic conventions for that language. In the case of verbs, the lemma is typically the infinitive form or, alternatively, the present tense of the verb (note that if an external particle is used to indicate the infinitive as in English "to play", this particle should be omitted). For nouns it is the noun singular form, while for adjectives it is the positive (i.e., non-negative, non-graded) form. For multi-word entries it is assumed that the same principles of lemmatization are applied to the head word.

The property canonical form has a Lexical Entry as domain and a Form as range. It is a subproperty of the lexicalForm property. The canonical form has to be unique, such that the property canonicalForm is modelled as functional:

canonicalForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/canonicalForm

The canonical ("dictionary") form of the lexical entry. This usually indicates the "lemma" form of a lexical entry

Domain: LexicalEntry

Range: Form

Characteristics: Functional

SubPropertyOf: lexicalForm

The lexical entries with the canonical form the noun "cat", the verb "marry" and the adjective "high" would look as follows (in Turtle syntax):

:lex_cat a ontolex:LexicalEntry;
         ontolex:canonicalForm :form_cat.
    
    :form_cat ontolex:writtenRep "cat"@en .
    
    :lex_marry a ontolex:LexicalEntry;
         ontolex:canonicalForm :form_marry.
    :form_marry ontolex:writtenRep "marry"@en .
    
    :lex_high a ontolex:LexicalEntry;
         ontolex:canonicalForm :form_high.
    :form_high ontolex:writtenRep "high"@en .

Of course, lexical entries need not to correspond to one word only, they can correspond to a multi-word term, as the following example for the lexical entry "intangible assets" shows:

:lex_intangible_assets a ontolex:LexicalEntry;
         ontolex:canonicalForm :form_intangible_assets.
    
    :form_intangible_assets ontolex:writtenRep "intangible assets"@en .

We will see below how the lexical entry can be decomposed in its parts using the machinery introduced by the decomposition module.

It is also possible to indicate non-canonical forms of lexical entries which we call "other forms":

otherForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/otherForm

A non-preferred ("non-dictionary") representation of a lexical entry. This should be a variant that is either a morphological variant, an abbreviation, short form or acronym

Domain: LexicalEntry

Range: Form

SubPropertyOf: lexicalForm

For example we may specify non-canonical forms of the verb ""marry"" as follows:

:lex_marry a ontolex:LexicalEntry ;
      ontolex:canonicalForm ex:form_marry ;
      ontolex:otherForm ex:form_marries .
    
    :form_marry ontolex:writtenRep "marry"@en .
    :form_marries ontolex:writtenRep "marries"@en .

We also define a property lexicalForm which subsumes both the canonical and other form property:

lexicalForm (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/lexicalForm

Indicates a single morphological form of an entry

Domain: LexicalEntry

Range: Form

Semantics

The model supports the specification of the meaning of lexical entries with respect to a given ontology. The lexicon model for ontologies follows the paradigm of semantics by reference1 in the sense that the meaning of a lexical entry is specified by pointing to the ontological concept that captures its meaning.

The property used for this is the property denotes, which is defined as the relation between a lexical entry and the logical predicate in an ontology that represents its meaning and has some denotational or model-theoretic semantics.

For the above mentioned lexical entries, this would look as follows:

:lex_cat a ontolex:LexicalEntry;
       ontolex:canonicalForm :form_cat;
       ontolex:denotes <http://dbpedia.org/resource/Cat>.
    :form_cat ontolex:writtenRep "cat"@en.
    
    :lex_marry a ontolex:LexicalEntry;
       ontolex:canonicalForm :form_marry;
       ontolex:denotes <http://dbpedia.org/ontology/spouse>.
    :form_marry ontolex:writtenRep "marry"@en .
    
    :lex_high a ontolex:LexicalEntry;
       ontolex:canonicalForm :form_high;
       ontolex:denotes <http://dbpedia.org/ontology/elevation>.
    :form_high ontolex:writtenRep "high"@en . 
    
    :lex_intangible_assets a ontolex:LexicalEntry;
      ontolex:canonicalForm :form_intangible_assets;
      ontolex:denotes <http://dbpedia.org/resource/Intangible_asset>.
    :form_intangible_assets ontolex:writtenRep "intangible assets"@en .

The property denotes is defined as follows:

denotes (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/denotes

The relation between a lexical entry and the logical predicate in an ontology that represents its meaning and has some denotational or model-theoretic semantics

Domain: ontolex:LexicalEntry

SubPropertyOf: semiotics:denotes

PropertyChain: sense o reference

The following example shows how we can model the fact that a certain lexical entry is ambiguous and assign it two (disjoint) meanings it can denote.

:lex_bachelor a ontolex:LexicalEntry;
       ontolex:denotes <http://dbpedia.org/resource/Bachelor>;
       ontolex:denotes <http://dbpedia.org/resource/Single_person>.
    
    :lex_bachelor ontolex:canonicalForm :form_bachelor.
    
    :form_bachelor ontolex:writtenRep "bachelor"@en .

No range is given for the denotes property, for two reasons. Firstly, the target of a denotation does not need to be an individual in the ontology but may also refer to a class, property or datatype defined by the ontology, as such it is not possible to specify the range of denotes in OWL. Secondly, the model is agnostic with respect to the ontology language used to express the ontology referred to, the assumption is is that the entity in the range represents some predicate that has a denotational semantics in some formal logical system.

All object properties in the model have an inverse property named as "is x-ed by", where x is the original property name. In the case of #denotes this property is #isDenotedBy.

isDenotedBy (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/isDenotedBy

The inverse relation of #denotes</div><div class='description'>denotes

Lexical Sense & Reference

For many practical modelling situations, the denotes property is not sufficient to capture the precise linking between a lexical entry and its reference and as such we introduce an intermediate element called lexical sense to capture the sense(s) of a word which refers to the ontology entity. The lexical entry is linked to a lexical sense by means of the sense property and the lexical sense is linked to the ontology by means of the reference property. LexicalSense (Class)

URI: http://www.w3.org/nl/lemon/ontolex/LexicalSense

Represents the lexical meaning of a lexical entry when interpreted as referring to the corresponding ontology element. A link between a lexical entry and an ontology entity via a LexicalSense object implies that the lexical entry can be used to refer to the ontology entity in question under corresponding connotations, pragmatic implications and meaning nuances that might no be directly modeled in the ontology. These pragmatic conditions and meaning nuances can be attached to the LexicalSense object, which represents a reification of the pair of LexicalEntry and ontology element. The Lexical Sense thus uniquely determines a lexical entry and an ontological entity it is associated with.

SubClassOf: ontolex:reference exactly 1 owl:Thing; ontolex:isSenseOf exactly 1 ontolex:LexicalEntry, semiotics:Meaning

A property definition can be used to attach a definition to the Lexical Sense; such a definition is often referred to as gloss.

definition (Datatype Property)

URI: http://www.w3.org/nl/lemon/ontolex/definition

This property relates a LexicalSense to one of its defintions; multiple definitions are allowed.

Domain: LexicalSense

Range: owl:Thing

Lexicons

Lexicon is an object that groups all lexical entries. A lexicon is specific for a particular language. The lexicon comprising the example entries we have created above would simply look as follows:

:lexicon a ontolex:Lexicon;
       ontolex:language "en";
       ontolex:entry :lex_high;
       ontolex:entry :lex_cat;
       ontolex:entry :lex_marry;
       ontolex:entry :lex_intangible_assets.

URI: http://www.w3.org/nl/lemon/ontolex/Lexicon

A collection of lexical entries specific to the given language and/or domain

entry (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/entry

This property links a lexicon to one of the lexical entries contained in it.

Lexical Concept

We have seen above how to capture the fact that a certain lexical entry can be used to denote a certain element in the extension of a class. We capture this by saying that the lexical entry denotes the class or ontology element in question.

However, sometimes we would like to express the fact that a certain lexical entry evokes a certain mental concept rather than that it refers to a class with a formal interpretation in some model.

For this, in the lexicon model for ontologies we introduce the class of LexicalConcept that represents ``a mental abstraction, concept or unit of thought that can be lexicalized by a given collection of senses''. As such. LexicalConcept is a sublcass of skos:Concept.

LexicalConcept (Class)

URI: http://www.w3.org/nl/lemon/ontolex/LexicalConcept

A mental representation or unit of thought that is expressable in some language.

SubClassOf: skos:Concept

The lexical entry is said to evoke a particular concept, similar to how a lexical entry denotes an ontology reference.

evokes (Object Property)

URI: http://www.w3.org/nl/lemon/ontolex/evokes

The relation between a lexical entry and the lexical concept it evokes, i.e. the mental concept that speakers of a language might associate when hearing the lexical entry.

Domain: Lexical Entry

Syntax and Semantics (synsem)

Most words in a language do not stand by their own, but have a certain syntactic behaviour in the sense that they appear in certain structures and require a number of syntactic (and semantic arguments). Examples of this are transitive verbs (e.g. to own), which require a syntactic subject and a syntactic object as well as relational nouns (e.g. capital (of), mother (of), son (of), brother (of), etc.) which require a prepositional object or adjectives which require a noun to modify. Frames are defined as follows:

Frame (Class)

URI: http://www.w3.org/nl/lemon/synsem/Frame

A (syntactic) frame is a linguistic object that represents the syntactic behaviour of an open class word in terms of (syntactic) arguments it requires. It essentially describes the subcategorization structure of the open class word in question, in particular the (syntactic) arguments it requires.

In order to specify the syntactic behaviour of lexical entries in terms a number of possible (syntactic) frames, the synsem module defines the property synBehaviour.

synBehaviour (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/synBehavior

The property syntacticBehaviour relates a LexicalEntry to one of its possible syntactic behaviours as represented by a #Frame</div><div class='description'>frame

The following snippet shows how to indicate that the verb (to) own can be used as a transitive verb. This is accomplished by adding a frame own_fram that is declared as a transitive frame, using the class http://lexinfo.net/ontology/2.0/lexinfo.owl#TransitiveFrame TransitiveFrame defined in the http://lexinfo.net/ontology/2.0/lexinfo.owl LexInfo Ontology.

      synsem:synBehavior :own_frame .
    
    :own_frame a synsem:Frame, lexinfo:TransitiveFrame.
    
    :own_form ontolex:writtenRep "own"@en .

A syntactic frame is defined by a number of (syntactic) arguments that need to be filled so that the frame is complete. A (syntactic) argument is defined as follows:

Argument (Class)

URI: http://www.w3.org/nl/lemon/synsem/Argument

An argument represents a slot that needs to be filled for a certain syntactic structure/projection of a lexical entry or a semantic frame to be complete. Syntactic arguments typically realise a certain grammatical function (e.g. subject, direct object, indirect object, prepositional object, etc.); semantic arguments represent an entity that plays some role in a semantic frame.

The object property synArg is used to relate a (syntactic) frame to one of its (syntactic) arguments

synArg (Object Property)

URI: http://www.w3.org/nl/lemon/synsem/synArg

Domain: Frame

From the semantic perspective, a (syntactic) frame corresponds to a semantic frame, defined as ``as a coherent structure of related concepts that are related such that without knowledge of all of them, one does not have complete knowledge of any one; they are in that sense types of gestalt.'' Decomposition (decomp)

Decomposition is the process of indicating which elements constitute a multi-word or compound lexical entry. The simplest way to do this is by means of subterm property, which indicates that a lexical entry is a part of another entry.

subterm (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/subterm

Indicates that one lexical entry is a constituent part of another lexical entry

Domain: LexicalEntry

Range: LexicalEntry

For example we may indicate that "landing" is a part of the lexical entry "amphibious landing" as follows:

:AmphibiousLanding a ontolex:LexicalEntry ;
      decomp:subterm :Landing.

The use of sub-terms has two principle limitations, firstly it is not possible to indicate inflection within a particular phrase of its individual elements, and secondly that it is not clear if the components represent the complete phrase or their relative order. To fix this we introduce an object called a component defined as follows

Component (Class)

URI: http://www.w3.org/nl/lemon/decomp/Component

A component represents a part of a lexical entry, that is either a word, sequence of words, phrase, or affix

Each component is said to be a constituent of a lexical entry

Constituent (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/constituent

Indicates a component that is part of an entry

Domain: LexicalEntry or Component

Range: Component

Furthermore, each of these components is said to identify a lexical entry, that gives the link to the word itself

Identifies (Object Property)

URI: http://www.w3.org/nl/lemon/decomp/identifies

Indicates that the component represents the usage of this particular lexical entry within another lexical entry

Domain: Component

Range: LexicalEntry or Argument

As such we may represent simple decompositions such as tokenizations as follows

:DesembarcoAnfibio a ontolex:LexicalEntry ;
      decomp:constituent :DesembarcoAnfibio_Desembarco_Component; 
      decomp:constituent :DesembarcoAnfibio_Anfibio_Component .
    
    :DesembarcoAnfibio_Desembarco_Component a decomp:Component ;
      decomp:identifies :Desembarco .
    
    :DesembarcoAnfibio_Anfibio_Component a decomp:Component ;
      lexinfo:gender lexinfo:masculine ;
      lexinfo:number lexinfo:plural ;
      decomp:identifies :Anfibio .

If we wish to specify the order of the tokenization of a lexical entry, we do this by making it an rdf:Seq whose elements are the words. As such the first token is indicated with the rdf:_1 special property and so forth, in addition to constituent. For example, we may indicate the tokens of "African Swine Fever" as follows:

      decomp:identifies :African .

Variation & Translation (vartrans)

We further consider TeminologicalVariants, which are defined as follows:

TerminologicalVariant (Class)

URI: http://www.w3.org/nl/lemon/vartrans/TerminologicalVariant

Terminological variants are variants that are semantically related in the sense that they can be exhcanged in most contenxt, but their surface forms are not directly related. The variants vary along dimensions that are not captured by the given ontology and are intentionally (pragmatically) caused.

SubclassOf: vartrans:source exactly 1 ontolex:LexicalSense; vartrans:target exactly 1 ontolex:LexicalSense; vartrans:Variant

Examples of terminological variants are:

  • Diatopic (dialectal or geographical variants) (e.g., gasoline vs. petrol)
  • Diaphasic (register) (e.g., headache and cephalalgia; swine flu and pig flu and H1N1 and Mexican pandemic flu)
  • Diachronic (or chronological variants) (e.g., tuberculosis and phthisis)
  • Diastratic (discursive or stylistic variâants) (e.g., man vs. bloke)
  • Dimensional variants: the terms point to the same concept but highlight a different

property or dimension of the concept (e.g., biosanitary waste vs. hospital waste; Novel Coronavirus vs. Middle East Respiratory Syndrome Coronavirus; obsolete technology vs. dangerous technology; madre de alquiler (rental mother) vs. vientre de alquiler (womb mother), in Spanish).

Translation

A special type of Variant is a variant that relates lexical senses across languages. We call such a variant Translation . Special types of such Translations are Direct Equivalents (representing terms that describe (highly) semantically equivalent entities that refer to entities that exist in both cultures and languages. E.g. surrogate mother in English, madre de alquiler in Spanish and mÃ¨re porteuse in French.) and Cultural Equivalents (representing the closest equivalent -or counterpart- of a sense in another language. Relation to SKOS

This part of the document is not part of the lexicon ontology model proper, but contains concrete recommendations and best practice patterns for combining SKOS models with the lexicon ontology model.

The use case we adress is one where a thesaurus or other taxonomic resource or classification system in SKOS needs to be enriched with more detailed linguistic information.

Imagine we have two concepts "financial assets" and "tangible assets" modeled in a SKOS. The official way to link a lexical entry to a SKOS concept (and viceversa) is through the property evokes (and its inverse isEvokedBy). The following example shows how to do this:

:financial_assets a skos:Concept;
                    ontolex:isLexicalizedBy :financial_assets_lex.
    
    :financial_assets_lex a ontolex:LexicalEntry;
                     ontolex:evokes :financial_assets;
                     ontolex:form :financial_assets_form. 
    
    :financial_assets_form ontolex:writtenRep "financial assets".

The above represents the standard way of linking a SKOS concept to a lexical entry in the lexicon ontology model.

To show how to make statements about preferred lexicalizations akin to the properties prefLabel, altLabel and hiddenLabel as used in SKOS, we give the following example:

:tuberculosis a skos:Concept;
         ontolex:isEvokedBy :tuberculosis_lex;
         ontolex:isEvokedBy :consumption_lex.
    
    :tuberculosis_lex a ontolex:LexicalEntry;
          ontolex:sense :tuberculosis_sense;
          ontolex:evokes :tuberculosis.
       
    :tuberculosis_sense a ontolex:LexicalSense;
          ontolex:isContainedIn :tuberculosis; 
          ontolex:context "preferred".
    
    :consumption_lex a ontolex:LexicalEntry;
           ontolex:sense :consumption_sense;
           ontolex:evokes :tuberculosis.
    
    :consumption_sense a ontolex:LexicalSense;
            ontolex:isContainedIn :tuberculosis;
            ontolex:context "outdated".

In case you are using reified labels as in SKOS-XL, it is possible to have Forms or LexicalEntries in the range of the skosxl:prefLabel, skosxl:altLabel and skosxl:hiddelLabel properties. However, we note that from this it would follows that LexicalEntries and Forms would be inferred to be skosxl:Labels, which does not correspond to the understanding of Forms and LexicalEntries of this community as linguistic objects rather than mere `labels'.
